Content-addressable storage (SHA-256 dedup)
Each chunk hashed with SHA-256. Identical chunks share one physical copy. ref_count tracks how many files reference a chunk. Storage savings of 30-50% typical in enterprise.
Chunked resumable uploads
Files split into 4-8MB chunks client-side. Each chunk uploaded independently with retry. Resumable from last successful chunk after network failure. Critical for mobile on spotty connections.
Hierarchical ACL with inheritance
Permissions flow from parent folder to children. Explicit permission overrides inherited. ACL evaluated bottom-up: file own permission > parent folder > grandparent. Cached in Redis for fast checks.
Optimistic concurrency (ETags)
Every file metadata update includes ETag (version hash). PUT /files/{id} with If-Match header. 412 Precondition Failed if stale. No pessimistic locks on metadata reads.
Eventual consistency for sync
Desktop/mobile clients sync via cursor-based delta feed. Changes propagate within seconds but not guaranteed instant. Conflict resolution: last-writer-wins for metadata, fork for content conflicts.
Append-only audit trail
Every file operation logged to immutable activity_log (Cassandra). Never UPDATE or DELETE audit rows. Enables compliance (SOC2, HIPAA), admin investigation, and undo operations.
02 Architecture Click components for deep dives
L1 — Client & Edge
CDN + API Gateway + Auth + Rate Limiting
>
🌐Mobile / Web Client
Chunked uploader + local sync engine
Client-side file chunking (4-8MB). SHA-256 per chunk for dedup check before upload. SQLite for local metadata cache. Exponential backoff on retry. Bandwidth throttling.
🚪API Gateway
Auth + Rate limit + Request routing
OAuth2 / JWT validation. Per-user + per-IP rate limits. Routes to appropriate service based on path prefix. TLS termination. Request logging.
⚡CDN (CloudFront/Akamai)
Download acceleration + edge caching
Pre-signed URLs for downloads. Edge caching for popular shared files. Geographic distribution reduces TTFB. Cache invalidation on file update via purge API.
L2 — Core Services
Metadata + Blob + Sharing + Sync + Search + Collab
>
📄File Metadata Service
CRUD + tree structure + ownership
Manages file/folder hierarchy, names, sizes, MIME types, ownership. PostgreSQL sharded by owner_id. ETag-based optimistic concurrency. Soft delete with 30-day retention.
📦Blob Storage Service
Chunked upload/download + dedup
Handles resumable uploads (initiate, upload chunk, finalize). SHA-256 dedup: check chunk hash before storing. Pre-signed URL generation for direct S3 upload/download.
👥Sharing & Permission Service
ACL + inheritance + link sharing
Hierarchical permission model: OWNER > EDITOR > COMMENTER > VIEWER. Inheritance from parent folders. Public/org-wide link sharing with optional password and expiry.
🔄Sync Service
Delta sync + conflict resolution
Cursor-based change feed. Clients poll with last cursor, receive only deltas. Long-polling or SSE for real-time push. Last-writer-wins for metadata; content conflicts create fork.
🔍Search Service
Full-text + metadata search (Elasticsearch)
Indexes file names, content (OCR for images, text extraction for docs), owner, shared-with, MIME type. Elasticsearch cluster. Permission-filtered results at query time.
📝Collaboration Service
Real-time co-editing + OT/CRDT
WebSocket connections for real-time editing. Operational Transform (Google Docs-style) or CRDT for conflict-free merges. Cursor position broadcast. Presence indicators.
L3 — Storage Layer
S3/GCS + PostgreSQL + Redis + Kafka
>
🗄Object Store (S3/GCS)
Blob storage + 11 nines durability
Stores file chunks keyed by content hash. Multi-tier: hot (S3 Standard), warm (S3 IA), cold (Glacier). Lifecycle policies auto-tiering based on access patterns.
💾Metadata DB (PostgreSQL)
Sharded by owner_id + read replicas
Files, versions, permissions, sharing links tables. Sharded by owner_id for data locality. 3-5 read replicas per shard for search/listing queries. PgBouncer for connection pooling.
⚡Cache (Redis Cluster)
ACL cache + metadata hot path + rate limits
Permission results cached with TTL (ACL check is on every API call). File metadata for frequently accessed files. User quota counters. Rate limit sliding windows.
📨Message Queue (Kafka)
Event bus + async job dispatch
Topics: file-events, sync-events, notification-dispatch, audit-log. Partitioned by owner_id. At-least-once delivery. All consumers idempotent.
L4 — Async Layer
Notifications + Thumbnails + Virus Scan + Quota + Audit
>
🔔Notification Service
Email + Push + In-app notifications
Kafka consumer on file-events. Batches notifications (avoid flooding on bulk operations). Respects user notification preferences. Dedup by event_id.
🖼Thumbnail / Preview Generator
Async image/doc preview generation
Generates thumbnails for images, PDFs, docs, videos. Multiple sizes (small, medium, large). Cached in CDN. Triggered on file upload completion event.
🛡Virus Scanner
ClamAV + heuristic scan on upload
Every uploaded file scanned before ACTIVE status. Quarantine infected files. Scan is async but blocks file from being shared/downloaded until clear. Periodic re-scan of existing files.
📊Quota Enforcement
Storage limits + billing integration
Pre-check before upload initiation. Atomic counter in Redis. Eventual consistency with actual storage (reconciliation job). Dedup-aware: shared chunks counted once per user.
📜Audit Logger
Immutable activity log (Cassandra)
Every file operation: create, read, update, delete, share, unshare. Append-only Cassandra table. Partitioned by owner_id + month. Compliance: SOC2, HIPAA, GDPR audit requirements.

05 Data Flows Step-by-step for critical paths
F-01
Resumable File Upload
Chunked + dedup + virus scan
>
01
Client
Split file into 4-8MB chunks. Compute SHA-256 per chunk and full file. Send POST /api/v1/uploads/resumable with file_id + total_size + checksum.
Client-side chunking enables resume on failure
02
Quota Service
Check user quota in Redis: INCRBY user:{id}:quota {total_size}. If exceeds limit, reject with 413. Atomic increment prevents race condition on concurrent uploads.
Pre-check avoids wasted upload bandwidth
03
Blob Service
For each chunk: check if SHA-256 hash exists in chunks table. If exists: increment ref_count (dedup hit, skip upload). If not: upload to S3, insert chunk record.
30-50% storage savings from dedup
04
Metadata Svc
All chunks received: verify full-file SHA-256 matches declared checksum. Update file status UPLOADING -> PROCESSING. Insert file_chunks mapping rows. Publish FILE_UPLOADED to Kafka.
Checksum mismatch = reject + cleanup chunks
05
Virus Scanner
Kafka consumer picks up FILE_UPLOADED event. Download file from S3. Run ClamAV scan. Clean: update status PROCESSING -> ACTIVE. Infected: set QUARANTINED, notify user.
Blocks sharing/download until scan completes
06
Thumbnail Gen
Parallel to virus scan: generate thumbnails (small/medium/large) for supported types. Upload to CDN-backed bucket. Update file metadata with thumbnail URLs.
Async, non-blocking for file availability
F-02
File Download
CDN + pre-signed URL + Range support
>
01
API Gateway
Authenticate user. GET /api/v1/files/{fileId}/content. Rate limit check. Forward to Blob Service.
JWT validation at edge, no DB lookup
02
Permission Svc
Check ACL: does user have VIEWER+ permission on this file? Check Redis cache first (TTL 5min). Cache miss: query permissions table, walk up folder hierarchy for inherited perms.
ACL check on every download request
03
Blob Service
Generate pre-signed S3/GCS URL (TTL=1h). If file is popular (CDN-cached): redirect to CDN edge URL. If Range header present: generate range-specific pre-signed URL.
302 redirect avoids proxying bytes through app
04
CDN Edge
Serve from edge cache if hit. Cache miss: fetch from S3 origin, cache at edge. Reassemble chunks if needed. Support partial content (206) for video streaming.
p99 TTFB <500ms with CDN
05
Audit Logger
Async log: FILE_DOWNLOADED event with user_id, file_id, IP, timestamp. Written to Cassandra audit log. Used for compliance reporting and admin investigation.
Non-blocking, fire-and-forget via Kafka
F-03
Share File with User
ACL creation + notification + inheritance
>
01
Permission Svc
Verify caller has OWNER or EDITOR role. Check if target user already has permission (deduplicate). If sharing a folder: permission will be inherited by all children.
Only OWNER can grant OWNER role
02
Metadata DB
INSERT INTO permissions (file_id, grantee_id, role, inherited). Update file.shared=true flag. Invalidate Redis ACL cache for this file and all children (if folder).
Cache invalidation is recursive for folders
03
Notification
Publish PERMISSION_GRANTED to Kafka. Notification service sends email with file link + custom message. In-app notification badge for target user. Push notification on mobile.
Respect user notification preferences
04
Sync Service
File appears in target user's "Shared with me" view. Sync engine pushes change to target user's connected clients via cursor update. File metadata sent as delta.
Near-instant appearance on target devices
F-04
Delta Sync
Client polls + applies deltas + conflict resolution
>
01
Client
POST /api/v1/sync/changes with last cursor. Long-poll (60s timeout). If no changes: server holds connection until change occurs or timeout. On timeout: reconnect immediately.
Long-polling reduces unnecessary requests
02
Sync Service
Query change_log table for entries after cursor. Filter by user's accessible files (ACL). Return batch of deltas: created, modified, deleted, moved, shared files.
Cursor is opaque token encoding timestamp + sequence
03
Client
Apply deltas to local SQLite database. For new/modified files: download content. For deletes: move to local trash. For moves: update local path. Update local cursor.
Local DB enables offline access
04
Conflict Resolver
If local changes conflict with server (same file, different edits): create conflict copy "filename (conflict copy).ext". Both versions preserved. User picks winner manually.
Never silently overwrite user data
F-05
Real-time Collaboration (5 users editing one doc)
WebSocket + OT transform pipeline + server authority + convergence
>
01
Client
User opens document. Client requests a one-time collaboration ticket from Auth Service (scoped to doc_id, expires in 30s). Establishes WebSocket to Collaboration Service at ws://collab.example.com/docs/{doc_id}?ticket={token}. Server validates ticket, returns the current document snapshot + server revision number + list of active user cursors.
Ticket prevents replay attacks; WebSocket for sub-50ms round-trip
02
Collab Service
Register user in the in-memory session for this document. Session tracks: (a) canonical document state, (b) current server revision number, (c) ordered ops log since last checkpoint, (d) connected clients list. Broadcast presence event { type: "user_joined", user_id, display_name, color } to all other 4 clients via their WebSocket connections.
Each active document is pinned to one collab server via consistent hashing on doc_id
03
Client (typing)
User types "Hello" at position 42. Client creates operation: { type: "insert", pos: 42, text: "Hello", client_rev: 7, server_rev: 15 }. Client applies op to local document immediately (optimistic). Sends op to server. Client buffers: if another keystroke happens before server ACKs, it's queued locally (not sent yet). At most 1 unacknowledged op in flight at a time.
Optimistic apply = no typing lag. 1-op-in-flight rule simplifies OT
04
OT Engine (server)
Server receives op from Client A at server_rev=15, but server is now at rev 17 (two ops from other clients arrived first). Server must transform A's op against those 2 intervening ops: (1) Fetch ops [16, 17] from the ops log. (2) For each: transform(A_op, server_op) -> A_op'. Example: if op 16 was insert at pos 30, and A's insert was at pos 42, A's position shifts to 47 (30 < 42, so +5 for the 5 chars inserted). (3) After transforming against all intervening ops, apply A_op' to server state. Assign new revision 18. Append to ops log.
Transform = adjust positions so the intent is preserved despite concurrent edits
05
Broadcast
Server sends the transformed op to all OTHER connected clients (B, C, D, E): { type: "insert", pos: 47, text: "Hello", server_rev: 18, author: "user_A" }. Server sends ACK to Client A: { ack: true, server_rev: 18 }. Client A now knows its op is committed. If Client A has buffered ops, it sends the next one (with server_rev: 18).
ACK before next send = serialization point ensuring causal ordering
06
Remote Clients
Client B receives op at server_rev 18. But Client B has 1 pending local op that hasn't been ACKed yet. Client B transforms the incoming server op against its pending local op (client-side OT). Applies the transformed op to its local document. Result: Client B's document now reflects both its own pending edit AND Client A's edit, with positions correctly adjusted. All 5 clients converge to the same document state once all ops are ACKed.
Client-side OT handles the "both editing at the same time" case
07
Cursor Sync
Every 100ms (throttled), each client sends cursor position: { type: "cursor", pos: 47, selection_end: 52 }. Server broadcasts to other clients without transformation (cursors are ephemeral, best-effort). Displayed as colored cursor lines with user name labels. No persistence needed — cursors are purely in-memory.
Fire-and-forget; slight cursor jitter is acceptable UX
08
Persistence
Every 30 seconds OR every 100 ops (whichever comes first): Checkpoint routine runs. (1) Serialize current document state to blob storage (S3). (2) Write checkpoint record to DB: { doc_id, revision: 118, blob_url, timestamp }. (3) Truncate ops log: delete ops before revision 118 (they're now baked into the checkpoint). (4) Also write ops to a durable WAL (write-ahead log in Kafka) BEFORE applying them, so on server crash, ops between checkpoints can be replayed.
WAL + checkpoint = zero data loss even on crash between checkpoints
09
Session End
When all users disconnect (or after idle timeout of 5 min): Final checkpoint written. Session object evicted from memory. File metadata updated with latest version number and size. Kafka event FILE_MODIFIED published for sync service / search reindex / notifications. Subsequent opens load from the latest checkpoint.
Graceful handoff from real-time to at-rest storage

Design Decisions
  • Shard by owner_id: All of a user's files on the same shard. Folder listing = single-shard query. Cross-user operations (shared files) require scatter-gather but are rare.
  • parent_id as FK: Enables tree traversal. Moving a file = UPDATE parent_id (O(1) instead of path rename). Orphan detection via periodic integrity check.
  • ETag-based optimistic locking: No pessimistic SELECT FOR UPDATE needed. ETag = SHA-256(id + version + modified_at). Client sends If-Match on every PUT. 412 on stale.
  • Soft delete with TTL: trashed_at timestamp starts 30-day countdown. Background GC job deletes TRASHED files where trashed_at < now() - 30d. Enables user "undo".
  • version column: Auto-incremented on every content update. Used for sync cursor comparison and conflict detection. Never decremented.

Design Decisions
  • Content-addressable: PK = SHA-256 hash of chunk data. Same content = same key. Dedup is automatic: INSERT ON CONFLICT (content_hash) DO UPDATE SET ref_count = ref_count + 1.
  • ref_count for GC: When a file version is deleted, decrement ref_count for each chunk. When ref_count reaches 0: chunk is orphaned. GC job deletes S3 object + DB row after 7-day grace period.
  • Storage tiering: Hot (S3 Standard): accessed in last 30 days. Warm (S3 IA): 30-90 days. Cold (Glacier): >90 days. Lifecycle rules auto-transition based on last_accessed. Retrieval from cold adds 3-5h latency.

07 Real-time Collaboration Deep Dive OT protocol + server authority + convergence guarantees
Core Problem
5 users have the same document open. All typing simultaneously. Each keystroke must appear on all 5 screens in <100ms, and all 5 copies must converge to the exact same text — even if two users type at the same character position at the same instant.
Solution: Operational Transform (OT)
Every edit is an "operation" (insert or delete at a position). When concurrent ops conflict, a mathematical transform function adjusts positions so every user's intent is preserved. The server is the single source of truth that assigns a total ordering.
OT Walkthrough Concrete example with 2 concurrent edits
Initial State (server rev 10)
"The quick fox"
Both Alice and Bob see this text. Both start typing at the same time.
Alice's Edit (client-side)
Inserts "brown " at position 10
Op: insert(10, "brown ")
Her screen: "The quick brown fox"
Sent to server with server_rev: 10
Bob's Edit (client-side)
Inserts " jumps" at position 13 (end)
Op: insert(13, " jumps")
His screen: "The quick fox jumps"
Sent to server with server_rev: 10
Server Processing (OT Engine)
Step 1: Alice's op arrives first. Server is at rev 10. No transform needed. Apply insert(10, "brown "). Server state: "The quick brown fox". Assign rev 11. Broadcast to Bob.
Step 2: Bob's op arrives. His op says server_rev: 10, but server is now at rev 11. Bob missed op [11] (Alice's insert). Transform required.
Step 3: Transform Bob's insert(13, " jumps") against Alice's insert(10, "brown "). Since Alice inserted 6 chars at pos 10, and Bob's pos 13 >= 10, Bob's position shifts: 13 + 6 = 19. Transformed op: insert(19, " jumps")
Step 4: Apply transformed op. Server state: "The quick brown fox jumps". Assign rev 12. Broadcast transformed op to Alice.
Final Converged State (rev 12) — All Clients
"The quick brown fox jumps"
Both users' intents preserved. Alice added "brown", Bob added "jumps". OT adjusted Bob's position so both edits land correctly.

Server-side Write Path What happens on the server for each incoming operation
1
Receive & Validate
WebSocket message arrives. Validate: is user still authorized? Is server_rev a valid reference point (not too old)? If op references a rev older than the oldest op in the log, reject — client must re-sync from checkpoint.
2
Acquire Document Lock
Acquire in-memory mutex for this document (not a distributed lock — document is pinned to one server). This serializes all ops for the same doc. Lock held for microseconds (transform + apply is pure CPU, no I/O).
3
Transform Against Concurrent Ops
Fetch all ops from the ops log between client's server_rev and current server rev. Apply OT transform function iteratively: for each server op, transform the client op to account for it. This adjusts character positions to reflect edits the client hasn't seen yet.
4
Write to WAL (Kafka)
Before applying to in-memory state, append the transformed op to a durable write-ahead log (Kafka topic partitioned by doc_id). This ensures that if the server crashes, ops between checkpoints can be replayed. Kafka acks = op is durable.
5
Apply to In-Memory State
Apply transformed op to the canonical in-memory document. Increment server revision. Append op to the in-memory ops log (ring buffer). Release document lock.
6
Broadcast & ACK
Send ACK with new server_rev to the originating client. Send the transformed op (with author + server_rev) to all other connected clients. Non-blocking fan-out via async WebSocket writes.
Why All Clients Converge Three properties that guarantee consistency
1. Server Total Ordering
The server assigns a sequential revision to every operation. All clients see ops in the same server-determined order. Even if ops arrive out of order from different clients, the server serializes them. This is the single source of truth.
2. Transform Correctness (TP1)
The OT transform function satisfies the "Transformation Property 1": if ops A and B are concurrent, then apply(apply(state, A), T(B, A)) = apply(apply(state, B), T(A, B)). Both orderings produce the same final state. This is a mathematical invariant of the transform functions.
3. Causality Preserved
The 1-op-in-flight rule ensures causal ordering: a client won't send op B until op A is ACKed. Combined with server_rev tracking, each op is transformed against exactly the right set of concurrent ops — no more, no less.

OT Transform Rules How positions are adjusted for each op combination
Insert vs Insert
insert(posA, textA) vs insert(posB, textB)
If posA < posB: B.pos += len(textA)
If posA > posB: A.pos += len(textB)
If posA == posB: tie-break by user_id (lower ID goes first)
Insert vs Delete
insert(posA, text) vs delete(posB, lenB)
If posA <= posB: B.pos += len(text)
If posA > posB + lenB: A.pos -= lenB
If posA in [posB, posB+lenB]: split the delete around the insert
Delete vs Delete
delete(posA, lenA) vs delete(posB, lenB)
Non-overlapping: adjust positions normally
Overlapping: reduce length to avoid double-delete
Fully contained: one delete becomes no-op

09 Architectural Tradeoffs Staff-level decision framing
Strong vs Eventual Consistency for Sync
User experience
System complexity
Throughput
Eventual consistency wins. Strong consistency requires distributed locks across all devices. At 15M DAU, latency cost is prohibitive. Eventual with conflict resolution (fork + manual merge) is the industry standard (Dropbox, Google Drive).
Inline Dedup vs Async Dedup
Storage savings
Upload latency
Implementation effort
Inline dedup for uploads. Check SHA-256 before storing chunk. If match: skip upload entirely (save bandwidth + storage). 1-2ms hash lookup cost per chunk is negligible vs 100ms+ upload time saved. Async dedup only for cross-user detection of near-duplicates.
Flat Namespace vs Hierarchical Tree
User mental model
Move/rename perf
Permission inheritance
Hierarchical tree with parent_id. Users expect folders. Move = update parent_id (O(1)). Rename = update name (O(1)). Permission inheritance traverses tree bottom-up. Alternative (materialized path "/a/b/c"): move requires updating ALL descendant paths — O(N) writes for large folders.
OT (Operational Transform) vs CRDT (Conflict-free Replicated Data Type)
Server dependency
Offline support
Memory efficiency
Implementation maturity
OT for Google Docs-style editing. OT requires a central server (single point of serialization) but is proven at Google's scale, uses less memory (no tombstones), and has simpler intent preservation for text. CRDT shines for peer-to-peer/offline-first apps (Figma uses CRDTs) — no server needed for convergence, but document state grows with edit history (tombstone overhead). For a server-centric system like Google Drive where you always have connectivity, OT is the pragmatic choice. Alternative: hybrid — OT for online editing, CRDT for offline conflict resolution.
Server-side vs Client-side Encryption
Security guarantee
Feature compatibility
Search / preview
Server-side by default, client-side optional. Client-side encryption (E2EE) prevents server from reading content. But kills search, thumbnails, collaboration, virus scanning. Offer E2EE as enterprise option for sensitive folders. Server-side (AES-256) for everything else.
10 Pitfall Questions Expect these at Staff level interviews
"User A shares a folder with User B. User A then moves a file into that folder. Does User B see it? What if User A removes the file from the folder?"
  • Permission inheritance is dynamic. When the file is moved INTO the shared folder, it inherits the folder's permissions at query time. User B gains access immediately (after ACL cache invalidation).
  • Moving OUT revokes inherited access. The file's parent_id changes. ACL check walks up the NEW parent chain. If User B has no direct permission on the file, access is revoked.
  • Cache invalidation is critical. On any move operation: invalidate Redis ACL cache for the file AND the old/new parent folders. Stale cache = security vulnerability (user retains access they shouldn't have).
  • Notification timing: User B may get a "file shared" notification but by the time they click, the file has been moved out and they get 403. Edge case: show "file no longer accessible" message, not a raw error.
"Two users rename the same file at the same time. What happens?"
  • ETag-based optimistic locking prevents lost updates. Both users read file with ETag "v3". User A sends PUT with If-Match: "v3", succeeds, file becomes "v4". User B sends PUT with If-Match: "v3", gets 412 Precondition Failed.
  • Client retry logic: User B's client automatically re-fetches file metadata (now "v4" with User A's name). Shows conflict UI: "This file was renamed by User A to 'new-name'. Do you still want to rename it?"
  • Move race conditions are harder: Two users moving the same file to different folders. Both succeed their ETag check (both read v3 at the same time). Solution: version column increment must be atomic: UPDATE files SET parent_id=?, version=version+1 WHERE id=? AND version=3. Only one succeeds.
  • Circular move detection: User A moves folder X into folder Y while User B simultaneously moves Y into X. Each operation must check: "would this move create a cycle?" by traversing up the target's ancestor chain. Use advisory locks for cycle detection during concurrent moves.
"How do you handle permission inheritance cycles? E.g., Folder A shared with Group X, which includes User who owns Folder B, which is parent of Folder A."
  • Permissions and folder hierarchy are separate graphs. Permissions form a DAG (directed acyclic graph): file -> permissions -> users/groups. Folder hierarchy is a tree (enforced by cycle detection on move). These two graphs cannot create cycles with each other.
  • Group membership is a separate concern. User being in Group X doesn't affect folder hierarchy. Permissions are checked at access time: "does this user have access via any path?" (direct permission OR group membership OR inherited from parent folder).
  • Real danger: recursive sharing. Share folder A with User B. User B shares subfolder A/C with User D. User D shares A/C/E back with Group that includes original Owner. This is valid — each sharing relationship is independent. No cycle, just multiple access paths to the same content.
  • Performance concern: Deep permission chains (10+ levels of folder nesting with different sharing at each level). Solution: flatten effective permissions into Redis cache. Cache key = (file_id, user_id) -> effective_role. Recompute on any permission change in ancestor chain.
"With content-addressable dedup, how do you handle quota accounting? If 100 users all upload the same 1GB file, who gets charged?"
  • Logical vs physical storage are different. Physical: one copy of the 1GB file exists (ref_count=100). Logical: each user's quota is charged the full 1GB. This is the Google Drive / Dropbox model.
  • Why charge everyone: If User A deletes their copy, their quota should decrease. If dedup-aware: deleting would not free quota (other 99 users still reference it). Confusing UX: "I deleted a 1GB file but my storage didn't change."
  • Implementation: Quota counter (Redis INCRBY) tracks logical usage per user. File size added on upload, subtracted on delete. Completely independent of physical dedup. Reconciliation job periodically verifies Redis counter matches SUM(files.size) from DB.
  • Enterprise exception: Some enterprise plans offer "dedup-aware quota" where shared files within an org only count once against the org's pool. This is a billing feature, not a storage feature. Implemented as: org_quota = SUM(DISTINCT chunk sizes) for all org users.
"A user has a folder with 500,000 files. How do you make listing performant?"
  • Cursor-based pagination is mandatory. Never use OFFSET/LIMIT (O(N) for deep pages). Use keyset pagination: WHERE (name, id) > (:last_name, :last_id) ORDER BY name, id LIMIT 50. Cursor encodes the last item's sort key.
  • Index on (parent_id, name, id): Composite index makes folder listing a single index range scan. No table access needed if query only needs name + id + type (covering index).
  • Pre-computed counts: Don't SELECT COUNT(*) for folder item count (full scan). Maintain children_count column on folders, incremented/decremented on add/remove. Eventually consistent — reconciliation job fixes drift.
  • Client-side virtual scrolling: Even with fast backend, rendering 500K items in DOM is impossible. Client uses virtual list: only renders visible ~50 items. Fetches next page on scroll. Illusion of infinite list.
  • Background pre-fetch: When user navigates to a folder, pre-fetch first 2 pages. On scroll near bottom, prefetch next page. Eliminates perceived loading time for smooth scrolling experience.
"5 users are editing a Google Doc simultaneously. The collaboration server crashes mid-session. What happens to their unsaved edits? How do you recover without losing any keystrokes?"
  • WAL guarantees zero data loss. Every op is written to Kafka (partitioned by doc_id) BEFORE applying to in-memory state. Even if the server process is kill -9'd, all acknowledged ops exist in the WAL. Unacknowledged ops (in-flight at crash time) are the client's responsibility — the client will resend on reconnect.
  • Recovery: checkpoint + WAL replay. The document is reassigned to a new server (consistent hashing with virtual nodes). New server loads the last checkpoint (full document snapshot from blob storage, e.g., at revision 500). Then replays all ops from Kafka from revision 501 onward. Document state is reconstructed in <2 seconds.
  • Client reconnection. WebSocket disconnect triggers automatic reconnect with exponential backoff (100ms, 200ms, 400ms...). On reconnect, client re-authenticates with a new ticket, receives the current document snapshot + revision. Client diffs its local state against the server snapshot. Any pending/buffered ops that weren't ACKed are transformed against missed server ops and re-submitted.
  • Edge case: split-brain. If the old server isn't actually dead (network partition), two servers could accept ops for the same doc. Prevention: fencing token on the Kafka WAL. Each server generation gets a monotonic token. Kafka rejects writes from the old generation. Additionally: all clients disconnect on WebSocket error and re-resolve which server to connect to via the load balancer.
  • User experience: Users see a brief "Reconnecting..." banner (1-3 seconds). No keystrokes lost. Cursors of other users may temporarily disappear during recovery. "Last saved" indicator updates once final checkpoint is written post-recovery. For the user, it's a minor flicker, not a data loss event.
"What if 200 users open the same Google Doc at once for a company all-hands? Does OT scale to that many concurrent editors?"
  • OT itself scales — the bottleneck is broadcast fan-out. Transform is O(concurrent_ops_since_last_rev) per incoming op, which is typically 0-5 ops. Even with 200 users, at ~2 ops/sec/user = 400 ops/sec, each transform looks back at most a few ops. The CPU cost is negligible.
  • WebSocket fan-out is the real challenge. Each op must be broadcast to 199 other connections. At 400 ops/sec * 199 = ~80K WebSocket messages/sec from one server. Solution: batching (aggregate ops into 50ms windows, send one batch instead of 20 individual messages) + read-replica fan-out (primary handles OT, replicas handle broadcasting to subsets of clients).
  • Cursor spam reduction. 200 users * cursor updates every 100ms = 2000 cursor msgs/sec, each broadcast to 199 others = 400K msgs/sec. Mitigation: (a) Only broadcast cursor for users currently visible in the viewport of each recipient. (b) Throttle to 500ms for non-nearby cursors. (c) Show only ~8 cursors max (closest to your viewport position), others collapsed into "and 192 others..."
  • Document partitioning for extreme scale. If a doc has structured sections (like Google Slides or Sheets), partition OT by section/slide/sheet. Each partition has its own op log and can be handled by different servers. Text docs don't partition as cleanly, but paragraph-level partitioning is possible for 500+ user scenarios.
"How do you handle the case where a virus is found in a file that has already been shared with 1000 users?"
  • Immediate quarantine. Set file status = QUARANTINED. This blocks all downloads and sharing link access instantly. ACL check includes status check: status must be ACTIVE for any read operation.
  • Notify all affected users. Query permissions table for all users with access (direct + inherited). Send security notification: "A file shared with you was found to contain malware." Include file name, sharer, and action taken.
  • Audit trail investigation: Query activity_log for all FILE_DOWNLOADED events for this file_id. Generate report: which users downloaded it, when, from what IP/device. This report goes to the org's security team.
  • Sync engine handling: Push QUARANTINED status change via sync delta. Desktop clients that have local copies: mark file as quarantined in local DB, prevent opening, show warning. Do NOT auto-delete from local disk (user may need for forensic analysis).
  • Owner notification: File owner gets separate notification with option to: (a) delete the file permanently, or (b) appeal if they believe it's a false positive. Appeal triggers manual review by trust & safety team.
