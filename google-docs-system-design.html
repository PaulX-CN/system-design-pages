Availability Target
Structured document model
Documents represented as JSON/protobuf tree (paragraphs, styles, tables) — NOT binary file chunks. A 100-page Google Doc is one in-memory model (~1-5MB). OT operates on this semantic model.
Server-authoritative OT
Single serialization point per document. Server assigns global sequence numbers. All clients converge to the same state via transform functions. No split-brain — one truth.
One-operation-in-flight
Client buffers while awaiting ACK. Only 1 unacknowledged op at a time. Additional keystrokes composed into a single buffered op. Prevents transform explosion.
WAL + checkpoint
Every op written to Kafka WAL before applying. Periodic checkpoint (serialize document model to S3 every 30s/100 ops). On crash: load last snapshot + replay WAL. Recovery <2s.
Ephemeral presence
Cursor positions and selection state stored in Redis, not persisted. Fire-and-forget over WebSocket. Heartbeat every 30s for session liveness. 5min idle timeout.
Append-only audit trail
Every edit operation logged immutably. Enables: version history, undo/redo, compliance (SOC2, HIPAA), admin investigation. Never UPDATE or DELETE audit rows.

State Machines

Client OT State Machine

Each client maintains a local OT state machine that governs when operations can be sent to the server. The 1-op-in-flight rule ensures the server only transforms against one client op at a time.

S1
SYNCHRONIZED — Client and server are in sync. Local document matches server state at revision r. Any new local edit transitions immediately to S2.
S2
AWAITING_ACK — Client has sent one op to the server and is waiting for ack. During this state:
  • Receive ack → The server accepted our op. Update local server_rev. Transition → S1 (SYNCHRONIZED).
  • Receive remote_op → Another user's edit arrived. Transform our in-flight op against the remote op using OT. Stay in S2.
  • Local edit → User types more. Cannot send yet (1-op-in-flight). Buffer edits locally, compose into a pending op. Transition → S3.
S3
AWAITING_ACK + BUFFER — Client has one in-flight op AND a buffered pending op. During this state:
  • Receive ack → In-flight op accepted. Immediately send the buffered op as the new in-flight. Transition → S2.
  • Receive remote_op → Transform BOTH in-flight and buffered ops against remote op. Stay in S3.
  • Local edit → Compose new edit into the existing buffer. Stay in S3.
Why compose the buffer? — If the user types "abc" while awaiting ack, we don't queue three ops. We compose insert("a"), insert("b"), insert("c") into a single insert("abc"). This minimizes network round-trips and transform complexity.

Document Session Lifecycle

Each document in memory on the Collaboration Service goes through these states. This manages resource allocation and ensures graceful cleanup.

L1
LOADING — First user opens the doc. Service loads latest snapshot from S3, replays any ops from PostgreSQL since that snapshot, and reconstructs the in-memory document model. Timeout: 5 s → fail with DOC_LOAD_TIMEOUT.
L2
ACTIVE — One or more users are editing. The service holds the document model in memory, accepts ops via WebSocket, transforms and broadcasts them. Checkpoint triggers: every 30 seconds OR every 100 ops, whichever comes first.
L3
CHECKPOINTING — Serialize the in-memory model to S3, record the checkpoint in doc_snapshots, and truncate applied ops. Ops continue to be accepted during checkpoint (they queue). If checkpoint fails, retry up to 3 times, then force-evict with error.
L4
IDLE — All users disconnected but document is still in memory. Grace period: 5 minutes. If a user reconnects within this window, transition back to ACTIVE instantly (no S3 reload). After timeout → L5.
L5
CLOSED — Final checkpoint written, in-memory state evicted, WebSocket connections closed. Next access starts fresh from L1. The document's shard assignment is released for rebalancing.
Crash mid-ACTIVE — If the Collaboration Service crashes, the Kafka WAL has all ops since the last checkpoint. A standby node loads the snapshot + replays WAL ops to recover the document in <2 s. Connected clients receive a snapshot frame to re-sync.
05

Data Flows

F-01  Real-Time Collaborative Editing

End-to-end path of a single keystroke from one user to all collaborators. This is the critical hot path — every millisecond matters.

1
Local Apply — User types a character. The editor immediately applies the op to the local document model (optimistic update). UI reflects the change with zero latency.
2
Send Op — If in SYNCHRONIZED state, client sends {type: "op", rev: 42, ops: [insert("H", pos:7)]} over WebSocket. Transitions to AWAITING_ACK. If already in AWAITING_ACK, the op is composed into the buffer.
3
WAL Write — Collaboration Service receives the op and appends it to Kafka (WAL) before processing. This ensures durability — if the server crashes after this point, the op is recoverable.
4
Mutex Acquire — Per-document mutex acquired. Only one op is processed at a time for a given document. This serialization point is what makes server-authoritative OT work.
5
Transform — If the client's rev is behind the server's head, fetch all ops between client_rev and server_head from the ops log. Transform the incoming op against each one (OT). This produces the op that can be applied to the current server state.
6
Apply & Persist — Apply the transformed op to the in-memory document model. Write to doc_ops table with the new server_rev. Increment the server revision counter. Release mutex.
7
ACK Sender — Send ack with new server_rev back to the originating client. Client transitions from AWAITING_ACK → SYNCHRONIZED (or sends buffered op if in AWAITING_ACK+BUFFER).
8
Broadcast — Send remote_op to all OTHER connected clients on the same document. Each receiving client transforms the remote op against any local pending ops and applies it.
9
Checkpoint Check — If ops_since_snapshot ≥ 100 or last_checkpoint > 30 s ago, trigger an async checkpoint: serialize document model → S3, record in doc_snapshots, truncate old ops.
Latency budget — Steps 3-8 target <50 ms p99. WAL write (3) ~5 ms, mutex + transform (4-5) ~10 ms, persist (6) ~15 ms, network broadcast (7-8) ~10 ms. Total: ~40 ms typical.

F-02  Crash Recovery

When a Collaboration Service node crashes, connected clients lose their WebSocket. The system must recover the document state and reconnect clients within the 2 s SLA.

1
Failure Detection — Health checker detects the crashed node within 500 ms (missed 3 heartbeats at 150 ms interval). Marks the node as dead in the service registry.
2
Shard Reassignment — The document's shard is reassigned to a standby node. The standby node begins recovery for all documents that were active on the crashed node.
3
Snapshot Load — Load the latest checkpoint from S3 for each affected document. This gives us the document state as of the last checkpoint.
4
WAL Replay — Read all ops from Kafka (WAL) since the last checkpoint. Replay them in order against the loaded snapshot to reconstruct the exact server state at the time of crash.
5
State Verified — Validate the reconstructed document model (hash check against last known hash in ops log). Document is now ready to accept new connections.
6
Client Reconnect — Clients detect WebSocket disconnection and request a new ticket via POST /api/collab/ticket. Gateway routes to the new shard. Client receives a snapshot frame if needed.
7
Catch-Up — If the client has local unacknowledged ops (was in AWAITING_ACK or AWAITING_ACK+BUFFER), it re-sends them after connecting. The server transforms and applies as normal.
Recovery time budget — Detection: 500 ms + Shard reassignment: 200 ms + Snapshot load: 500 ms + WAL replay: 300 ms + Client reconnect: 500 ms = ~2 s total. Kafka retention for WAL: 24 hours (covers long outages). Checkpoint every 30 s ensures WAL replay is at most ~30 s of ops.

F-03  Version History & Restore

Users can view the edit history of a document and restore to any previous version.

1
Request History — Client calls GET /api/docs/{id}/history?granularity=auto. The server groups checkpoints into meaningful versions (auto-coalesces rapid edits into sessions).
2
Load Snapshot — When user selects a version, load that checkpoint from S3. Render as read-only preview with diff highlighting against the current version.
3
Restore — "Restore this version" creates a NEW set of ops that transform the current document into the target version. Active collaborators see the restore as normal incoming edits — no disruption.
4
Audit — Restore event logged to audit trail with who restored, from which version, and when. The previous state is still recoverable from the ops log.

Operational Transform Deep Dive

Walkthrough: Two Concurrent Edits

Alice and Bob are editing the document "ABCD" simultaneously. Alice inserts "X" at position 1, Bob deletes character at position 3. Both start from revision 5.

1
Initial State (rev 5): Document = "ABCD". Alice sees "ABCD", Bob sees "ABCD".
2
Alice types: insert("X", pos:1) → local doc becomes "AXBCD". Sends op to server with rev:5.
3
Bob deletes: delete(pos:3) → local doc becomes "ABC" (removed "D"). Sends op to server with rev:5. But Alice's op arrives first.
4
Server processes Alice's op: rev 5 matches server head → no transform needed. Apply insert("X", pos:1). Server doc = "AXBCD", rev → 6. ACK Alice, broadcast to Bob.
5
Server processes Bob's op: Bob's rev=5 but server is at rev=6. Need to transform Bob's op against Alice's op (rev 5→6).

Transform: Bob's delete(pos:3) vs Alice's insert(pos:1). Since Alice inserted BEFORE Bob's delete position, shift Bob's position right by 1. Transformed: delete(pos:4). Apply to "AXBCD" → "AXBC". Server rev → 7.
6
Both converge: Alice receives Bob's transformed delete(pos:4), applies to "AXBCD" → "AXBC". Bob receives Alice's insert("X", pos:1), transforms his pending ops, applies → "AXBC". Both see "AXBC" at rev 7. ✓ Convergence achieved.

Server Write Path (Pseudocode)

function handleOp(docId, clientOp, clientRev):
    mutex.acquire(docId)
    try:
        serverHead = getServerRev(docId)

        // Transform against all ops the client hasn't seen
        if clientRev < serverHead:
            missedOps = getOps(docId, from=clientRev, to=serverHead)
            for each serverOp in missedOps:
                clientOp = transform(clientOp, serverOp)

        // Apply to server document model
        doc = getDocModel(docId)
        doc.apply(clientOp)

        // Persist
        newRev = serverHead + 1
        writeOp(docId, newRev, clientOp)
        writeToKafka(docId, newRev, clientOp)  // WAL

        // Respond
        sendAck(clientId, newRev)
        broadcast(docId, clientOp, excludeClient=clientId)

        // Maybe checkpoint
        if shouldCheckpoint(docId):
            asyncCheckpoint(docId, doc, newRev)
    finally:
        mutex.release(docId)

Transform Rules

The transform(opA, opB) function takes two concurrent ops and returns a transformed opA' that achieves the same intent when applied after opB.

Op AOp BTransform Rule
insert(pos_a)insert(pos_b)If pos_a ≤ pos_b: no change. If pos_a > pos_b: pos_a' = pos_a + len(B)
insert(pos_a)delete(pos_b)If pos_a ≤ pos_b: no change. If pos_a > pos_b: pos_a' = pos_a - len(B)
delete(pos_a)insert(pos_b)If pos_a < pos_b: no change. If pos_a ≥ pos_b: pos_a' = pos_a + len(B)
delete(pos_a)delete(pos_b)If ranges overlap: shrink A to non-overlapping portion. If pos_a > pos_b: pos_a' = pos_a - len(B)
format(range_a)insert(pos_b)If insert inside range: expand range_a by len(B)
format(range_a)delete(range_b)If delete overlaps range: shrink range_a accordingly

Convergence Guarantee — TP1

Transformation Property 1 (TP1) states: for any two concurrent operations A and B, applying A then B' (B transformed against A) must produce the same document state as applying B then A' (A transformed against B).

apply(apply(doc, A), transform(B, A)) = apply(apply(doc, B), transform(A, B))
Why server-authoritative OT is simpler — TP1 is sufficient for correctness when there's a central server that serializes ops. Peer-to-peer OT requires the stronger TP2 property, which is much harder to satisfy. Google Docs uses server-authoritative OT precisely because TP1 alone is provably correct in this model.

Tradeoffs

OT vs CRDT

OT (our choice): Server-authoritative, proven at Google's scale since 2006. Simpler client logic (TP1 sufficient). Smaller op payloads. Requires central server per document.

CRDT: No central authority needed — true peer-to-peer possible. Better offline support (merge without server). But: larger metadata overhead (tombstones, vector clocks), harder to implement rich text formatting, less mature for production document editors.

Decision: OT. We already need a server for ACL, persistence, and presence. The "no server" advantage of CRDTs doesn't apply. OT's smaller wire format and proven track record in document editing wins.

Checkpoint Frequency vs Recovery Time

Frequent checkpoints (every 30s / 100 ops): Fast recovery (<2 s). Small WAL replay window. But: more S3 writes, more CPU for serialization, brief pauses during checkpoint.

Infrequent checkpoints (every 5 min / 1000 ops): Less S3 cost. But: longer WAL replay on crash (up to 1000 ops), slower recovery, larger ops table.

Decision: 30s / 100 ops. The 2s recovery SLA demands short WAL replay. S3 put cost is ~$0.005/1000 requests — negligible at scale. Checkpoint is async and doesn't block ops.

Document-Level vs Paragraph-Level OT

Document-level: Single ops log per document. All edits serialize through one mutex. Simpler to reason about. Sufficient for <50 concurrent editors.

Paragraph-level: Independent ops logs per paragraph. Concurrent edits to different paragraphs never conflict. Enables higher concurrency. But: structural ops (moving paragraphs, merging/splitting) require cross-paragraph coordination.

Decision: Document-level by default. Paragraph-level for documents with >50 active editors (auto-detected). The complexity of cross-paragraph structural ops isn't worth it for typical usage patterns.

WebSocket vs Server-Sent Events

WebSocket (our choice): Full duplex — client sends ops and receives remote ops on the same connection. Lower latency for rapid back-and-forth. Native binary frame support.

SSE + HTTP POST: Simpler infrastructure (works through HTTP proxies). Better for read-heavy use cases. But: two connections per client (SSE for receive, POST for send), higher latency for ops, no binary frames.

Decision: WebSocket. Real-time collaborative editing is fundamentally bidirectional with sub-50ms latency requirements. SSE would add unacceptable latency for the POST-based send path.

In-Memory vs Persistent Document Model

In-memory (our choice): Document model lives in RAM on the collab server. Transforms are instant. Checkpointed to S3 periodically. Risk: server crash loses in-memory state between checkpoints.

Persistent (Redis/DB): Every op immediately persisted to shared state. Any server can pick up any document. But: adds 5-10 ms per op for the persistence round-trip, which violates our <50 ms p99 target.

Decision: In-memory with WAL. The Kafka WAL gives us durability without the per-op latency penalty. Recovery from crash replays WAL ops — at most 30 s of ops, taking <500 ms to replay.

Structured Document Model vs Plain Text + Markup

Structured model (our choice): Document is a tree of typed nodes (paragraph, heading, list, table). OT ops are semantic: insert_text, apply_format, split_node. Preserves document structure during transforms.

Plain text + markup: Document stored as flat text with embedded markup (like Markdown). Simpler OT (just text insert/delete). But: formatting ops can corrupt markup, structure-changing ops (indent, merge cells) are nearly impossible to handle correctly.

Decision: Structured model. Rich text editing requires understanding document structure. Google Docs uses a similar tree-based model internally. The additional transform rules for structural ops are worth the correctness guarantee.

Pitfalls & Interview Questions

Q1: Server crashes mid-edit with 50 active users. How do you recover without losing any edits?

A
WAL + Checkpoint recovery. Every op is written to Kafka (WAL) before being applied. On crash: (1) standby node loads latest S3 snapshot, (2) replays all Kafka ops since that snapshot, (3) reconstructs exact server state. Clients reconnect and re-send any unacked ops. Zero data loss because WAL is durable before ACK. Recovery target: <2 seconds.

Key insight: The WAL write happens BEFORE the ACK. Any op the client thinks was accepted is guaranteed to be in Kafka. Any op that wasn't ACKed will be re-sent by the client after reconnection.

Q2: 200 users are editing the same document simultaneously. The OT server becomes a bottleneck. How do you scale?

A
Paragraph-level OT + connection offloading. Three-pronged approach:

1. Connection offloading: Separate WebSocket termination (fan-out servers, 100K conns each) from OT processing. Fan-out servers handle presence/cursor broadcasts without touching the OT server.

2. Paragraph-level OT: When editor count exceeds threshold (e.g., 50), automatically switch to paragraph-level mode. Each paragraph gets its own revision counter and ops log. Concurrent edits to different paragraphs never need transformation.

3. Read replica offloading: Of the 200 users, likely 150+ are viewers. Route viewers to read-only replicas that receive the op stream. Only editors connect to the OT server.

Q3: A user edits offline for 2 hours then reconnects. They have 500 unsynced operations. How do you merge without conflicts?

A
Transform chain + smart rebasing. The client cached its base revision (rev at disconnect). On reconnect:

1. Gap detection: Client sends its base_rev. Server sees the gap (e.g., client at rev 100, server at rev 600 — 500 remote ops happened).

2. If gap < 1000 ops: Server sends the 500 missed ops. Client transforms its 500 local ops against each of the 500 remote ops. This is O(n²) in the worst case but acceptable for <1000 ops.

3. If gap > 1000 ops: Server sends a fresh snapshot instead. Client must diff its local version against the snapshot and generate a minimal set of ops that represent the user's offline edits. These ops are then sent as a batch.

4. Conflict resolution: OT handles the merge mathematically. The user sees their offline edits applied cleanly alongside remote changes. If structural conflicts occur (e.g., both users deleted the same paragraph), the server's version wins and the client's edits are rebased.

Q4: Collaborative undo/redo — User A types "hello", User B types "world", then User A presses Ctrl+Z. What happens?

A
Selective undo, not global undo. Each user maintains their own undo stack of operations THEY performed. Ctrl+Z undoes only the user's own most recent op, not the global most recent.

Implementation: When User A presses Ctrl+Z, the client computes the inverse of their last op (insert "hello" → delete "hello"). But this inverse must be transformed against all ops that happened after the original — including Bob's insert "world".

Example: A inserted "hello" at pos 0, B inserted "world" at pos 5. Document: "helloworld". A presses undo. Inverse of A's op: delete(pos:0, len:5). Transform against B's insert(pos:5): since B's insert is after the delete range, no transform needed. Result: delete "hello", document becomes "world".

Edge case: If B's edit was INSIDE A's text (e.g., B formatted part of "hello" as bold), A's undo must delete "hello" but preserve B's formatting intent. This requires tracking edit provenance and falls back to "undo removes A's text but keeps B's modifications visible as a new standalone edit."
