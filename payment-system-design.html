Auth Validity Window
Idempotency at every layer
Every write is safe to retry. Gateway → Payment Service → PSP calls → Webhooks. Use idempotency_key + ON CONFLICT DO NOTHING at DB level.
State machine with optimistic locking
All status transitions use UPDATE … WHERE status='EXPECTED'. If 0 rows affected → concurrent transition won. Never use SELECT then UPDATE.
Double-entry ledger — immutable
Every debit has a corresponding credit. Sum of all signed entries = 0. Entries never updated — refunds create new reverse entries.
Transactional outbox
Kafka publish = side effect of DB transaction, not a parallel write. Solves dual-write atomically. payment update + outbox insert in same COMMIT.
Write PSP request log BEFORE calling
Log psp_requests row as PENDING before making external call. Enables crash recovery via reconciliation. Timeout ≠ Failure — query inquiry endpoint first.
Saga replaces 2PC
No distributed locks. Compensating transactions for rollback. If capture fails → issue fiat refund. If notification fails → retry idempotently.
UNKNOWN state is real
On PSP timeout: immediately set AUTHORIZATION_UNKNOWN or CAPTURE_UNKNOWN. Async job reconciles. Never show false success or failure to user.
Amounts in integer cents — NEVER float
0.1 + 0.2 ≠ 0.3 in IEEE 754. Store as BIGINT (cents). Never DECIMAL, never FLOAT. $100.00 = 10000. Currency field is ISO 4217 CHAR(3).
02 Architecture Click components for deep dives
Ingress Layer
Edge · Auth · Idempotency · Rate Limiting
▶
🚪API Gateway
Single ingress · TLS · Auth · Idempotency
OAuth2/HMAC auth. Per-merchant + per-IP rate limits. Idempotency key dedup via atomic INSERT ON CONFLICT. HMAC replay protection with ±5min window.
⏱️Rate Limiter
3 layers · Redis sliding window
Per-merchant · Per-IP · Global. Sliding window in Redis via atomic Lua (ZADD + ZREMRANGEBYSCORE + ZCARD). Token bucket for burst allowance.
🔁Idempotency Layer
Dedup · Cache · 24h TTL
INSERT INTO idempotency_keys ON CONFLICT DO NOTHING. 0 rows = duplicate → return cached response. COMPLETE after first successful response stored.
Core Services
Orchestration · Risk · Payment Methods · PSP
▶
📋Payment Service
State machine · Saga orchestrator
Owns the payment lifecycle. Optimistic locking on every state transition. Coordinates Risk, Payment Method, PSP, and Ledger services.
🛡️Risk & Fraud Engine
Rules (<10ms) + ML (<100ms)
Layer 1: deterministic rule engine (blacklists, velocity, BIN checks). Layer 2: gradient-boosted ML model. Decision: APPROVE/REVIEW/DECLINE + score 0–100.
🗄️Payment Method / Token Vault
PCI-DSS · Network-isolated · PAN tokenization
Raw PANs stored ONLY in isolated vault. App DBs store opaque token + card_fingerprint + last4. Vault is separate network segment, separate DB, separate deploy.
🔌PSP Integration Layer
Stripe · Adyen · Braintree · Failover
Auth → Capture → Settle lifecycle. Log psp_requests BEFORE calling. Hard vs transient retry logic. Circuit breaker: open at 5% error rate / 60s. Never change PSP mid-payment.
⚖️Chargeback / Dispute Service
Card network arbitration · SLA tracking
Manages full chargeback lifecycle with strict time windows. Auto-populates evidence from order data. Tracks chargeback rate — Visa/MC thresholds trigger escalating consequences.
🏦Settlement & Reconciliation
T+1/T+2 · PSP file matching · Exception handling
Daily PSP settlement files (CSV/SFTP). Match to internal records by psp_transaction_id. Categorize mismatches: timing lag, unmatched captures, fee discrepancies.
Financial Core
Double-entry ledger · Outbox · Events
▶
📒Ledger Service
Double-entry · Immutable · Source of truth
Every financial event = 2 entries. Sum of all entries always = 0. payment status + ledger insert + outbox event in single DB transaction. Never update — only append.
📤Transactional Outbox
Dual-write solved · Kafka consistency
outbox_events table written in same transaction as payment update. Outbox worker polls with FOR UPDATE SKIP LOCKED. Debezium CDC alternative at 5K+ TPS.
Data & Async Layer
Kafka · CQRS · Notifications · Webhooks
▶
📨Kafka Event Bus
At-least-once · Idempotent consumers
Topics: payment-events · ledger-events · webhook-dispatch · analytics. Partitioned by merchant_id. All consumers idempotent — payment_id + event_type as dedup key.
🔔Notification Service
Email · SMS · Push · Webhook

⚠️ UNKNOWN State — Critical Design Point
On PSP timeout: immediately set AUTHORIZATION_UNKNOWN or CAPTURE_UNKNOWN. Async reconciliation job polls PSP inquiry endpoint every minute. Never assume failure on timeout — the PSP may have succeeded. Show "Processing…" in UI. Resolve to SUCCESS or FAILURE only after PSP confirmation. Never retry the original call without querying inquiry endpoint first — could double-charge.
04 Complete Data Flows State-by-state · Exactly which tables are written
F-01
CREATED State — Payment Initiation
~5ms · DB write
▶
01
API Gateway
Validate JWT/HMAC. Check idempotency_key: INSERT ON CONFLICT DO NOTHING — 0 rows = duplicate, return cached response.
Atomic check-and-claim. No race condition possible.
02
Risk Engine
Rule engine (<10ms): blacklist, BIN check, velocity limits. ML model (<100ms): score 0–100. Score >70 → RISK_DECLINED (no DB write for payment).
Fast-path rejection before any DB write
03
Payment Service
BEGIN; INSERT payments (status=CREATED); INSERT payment_events (NULL→CREATED); INSERT outbox_events (PAYMENT_CREATED); COMMIT; — all or nothing.
Outbox in same transaction = no dual-write
04
Idempotency
UPDATE idempotency_keys SET status=COMPLETE, response='{id, status, ...}' WHERE key=?. Response cached for 24 hours.
Any retry within 24h returns this cached response
F-02
AUTHORIZING → AUTHORIZED (PSP Call)
~200–500ms · External call · Timeout risk
▶
01
Token Vault
Retrieve PAN from vault using payment_methods.card_token. Returns card details to PSP Integration Layer only — never persisted downstream.
PAN exists in memory only, never logged
02
PSP Layer
INSERT psp_requests (status=PENDING, idempotency_key=uuid) BEFORE calling PSP. Critical: this log enables crash recovery.
Write log BEFORE external call
03
Payment Service
UPDATE payments SET status=AUTHORIZING WHERE id=? AND status=RISK_APPROVED. If 0 rows → concurrent update won, abort.
Optimistic lock — no pessimistic SELECT FOR UPDATE
04
External PSP
Call stripe.authorize(amount, cardToken, idempotencyKey). Success → update psp_requests to SUCCESS, payments to AUTHORIZED. Timeout → set AUTHORIZATION_UNKNOWN immediately.
Timeout ≠ Failure. Never retry without inquiry.
05
Recon Job
If UNKNOWN: async job polls PSP inquiry API every 60s. stripe.retrieve(pspTransactionId) → resolve to AUTHORIZED or AUTHORIZATION_FAILED. Max 24h window.
Async resolution, not blocking the request path
F-03
CAPTURING → CAPTURED (Most Critical)
~300ms · Ledger + Outbox in same COMMIT
▶
01
PSP Layer
Check if capture already successful (idempotency). If auth expired (>7 days): must re-authorize. Log psp_requests BEFORE calling. Call stripe.capture(chargeId, amount).
Partial capture allowed: captured_amount ≤ authorized_amount
02
Payment Service
BEGIN; UPDATE payments SET status=CAPTURED (optimistic lock); INSERT ledger_entries × 2 (DEBIT customer_receivable / CREDIT merchant_payable); INSERT payment_events; INSERT outbox_events; COMMIT;
All 4 writes in ONE transaction — atomicity guaranteed
03
Outbox Worker
SELECT from outbox_events FOR UPDATE SKIP LOCKED. Publish PAYMENT_CAPTURED to Kafka. DELETE row. Downstream consumers: Notification, Webhook, Analytics, Settlement.
SKIP LOCKED = multiple workers, no contention
04
Settlement Job
T+1: PSP settlement file downloaded. Match by psp_transaction_id. Fee entries written to ledger: DEBIT psp_fees / DEBIT merchant_payable / CREDIT bank. Merchant payable reduced by PSP fee (2.9% + $0.30).
Settlement amount ≠ capture amount (PSP fees)
F-04
Chargeback Lifecycle
Days to weeks · Card network SLA windows
▶
01
Card Network
Cardholder disputes with issuing bank → Visa/MC initiates chargeback. Funds provisionally reversed from merchant. CHARGEBACK_RECEIVED event.
Time window starts — typically 7 days to respond
02
Ledger
DEBIT merchant_payable (reverse original credit). CREDIT chargeback_liability (new account type). Chargeback fee posted separately.
Real money movement — must hit ledger immediately
03
Evidence Engine
Auto-populate evidence package: order receipt, delivery confirmation, IP address, device fingerprint, prior successful transactions, customer communication history.
Auto-population critical — merchants won't respond manually at scale
04
Dispute Service
Submit evidence to card network via PSP API before deadline. Track response: WON (reverse chargeback, CREDIT merchant) or LOST (chargeback_liability becomes permanent).
SLA breach = auto-loss regardless of merits
05
Rate Monitor
Update chargeback_rate = chargebacks_this_month / transactions_last_month. Alert at 0.65% (Visa warning), block new onboarding at 0.9%, emergency at 1%.
Visa/MC use different denominators — track both
F-05
Refund Flow
~500ms · Reverse ledger entries · Never update old entries
▶
01
Payment Service
Validate: payment must be CAPTURED or SETTLED. refund_amount ≤ captured_amount - previously_refunded_amount. Set status PARTIALLY_REFUNDED or REFUNDED.
Partial refunds are common — track cumulative refunded_amount
02
PSP Layer
Call PSP refund API with idempotency key. PSP reverses original capture. Funds credited back to card. Timeline: 5–10 business days to cardholder.
PSP-level idempotency prevents double-refund on retry
03
Ledger
BEGIN; INSERT ledger_entries × 2 (CREDIT customer_receivable / DEBIT merchant_payable) — reverse of capture entries; INSERT outbox; COMMIT. NEVER update old entries.
Ledger is append-only — refunds create NEW entries

Design Decisions
  • BIGINT for cents, NEVER float. 0.1 + 0.2 ≠ 0.3 in IEEE 754. $100.00 = 10000. All arithmetic in integer domain. Currency conversion at settlement only.
  • version column for optimistic locking: UPDATE payments SET status=?, version=version+1 WHERE id=? AND version=? AND status='EXPECTED'. Rows_affected=0 → retry or fail.
  • psp_name locked at AUTH. Constraint: never change PSP mid-payment. Capture, refund, void all go to same PSP. Enforced at application layer.
  • Partial index on active payments: WHERE status NOT IN ('SETTLED','FAILED','REFUNDED','VOIDED'). Only ~0.01% of rows but queried constantly by ops dashboards.
  • Hot shard problem: Amazon/Shopify scale → millions of payments per hour. Virtual sharding: shard_key = hash(merchant_id || payment_id % 10) spreads one merchant across 10 shards. Dashboard queries scatter-gather 10 shards (async, acceptable).
  • Redis idempotency cache: key → {status, response_body} with 24hr TTL. Check Redis first before DB. Reduces DB reads for duplicate requests by ~90%.

3B rows/yr · ~600GB
Core Tables

idUUID PK

INVARIANT: SUM(CASE direction WHEN 'CREDIT' THEN amount ELSE -amount END) = 0 always. Verified by nightly reconciliation job — discrepancy = P0 alert.
NO UPDATE/DELETE: trigger RAISES EXCEPTION on any update attempt.
account_balance_snapshots
account_idUUID
balanceBIGINT
snapshot_atTIMESTAMP
last_entry_idUUIDcursor for delta query
Pattern: latest snapshot + sum of entries since snapshot_id. Avoids scanning full history. Snapshot every 1000 entries per account.
Design Decisions
  • Separate ledger DB from payments DB. Different write patterns, different backup/retention policies. Ledger entries have 7-year retention minimum. Payments can be archived earlier.
  • Trigger enforces immutability: CREATE TRIGGER prevent_ledger_mutation BEFORE UPDATE OR DELETE ON ledger_entries FOR EACH ROW EXECUTE FUNCTION raise_immutability_error();
  • FX entries: for multi-currency — store both original_amount + original_currency + settlement_amount + settlement_currency. FX rate locked at capture time. Reconcile settlement_amount against PSP file.
  • Chargeback accounts: new ledger account type CHARGEBACK_LIABILITY. Created when chargeback received. Reversed if won. Permanent if lost.
  • Reconciliation query: nightly job. SELECT SUM(CASE direction WHEN 'CREDIT' THEN amount ELSE -amount END) FROM ledger_entries. Must = 0. Any non-zero → P0 incident.
  • Read replicas for merchant balance queries. Write primary for all inserts. Replica lag <1s acceptable for balance display. Always read from primary for refund amount validation.

Design Decisions
  • psp_requests is 4TB/yr — largest table. Aggressive partitioning by month. After 90 days, compress with pg_compress. After 1 year, export to S3 Parquet + Athena for compliance queries.
  • request_payload stores full JSON of what was sent to PSP — critical for dispute resolution. "We sent exactly X amount with Y idempotency key." Regex/partial match on this field via Elasticsearch for fraud investigations.
  • Reconciliation process: (1) Download PSP CSV at 6AM. (2) Parse + normalize. (3) Match by psp_transaction_id. (4) Categorize: timing gap (auto-resolve), unmatched PSP record (alert), unmatched internal (alert), amount mismatch (alert). (5) Generate exception report.

Outbox & Events
PostgreSQL (tiny — delete after publish) Kafka (persistent event log)
<1M rows at any time
Core Tables
outbox_events — keep tiny
idUUID PK
payment_idUUID NOT NULL
event_typeVARCHARPAYMENT_CAPTURED etc.
payloadJSONB
statusVARCHARPENDING only (delete after)
retry_countINT DEFAULT 0
created_atTIMESTAMP
Index: (status='PENDING', created_at) — only index needed. Table stays tiny because rows deleted immediately post-publish.
Partitioning: hourly partitions for fast DROP PARTITION cleanup. At high TPS, individual DELETEs are expensive — DROP PARTITION is O(1).
FOR UPDATE SKIP LOCKED: allows multiple parallel workers without row contention.
Design Decisions
  • Outbox must be in same DB as payments. If in separate DB, you've reintroduced the dual-write problem. Same PostgreSQL instance = same COMMIT = atomic guarantee.
  • Delete vs mark PUBLISHED. Option A: DELETE after publish (keep table tiny, best performance). Option B: mark PUBLISHED (audit trail of every event). At scale, DELETE wins — table bloat kills performance. Kafka has the event log.
  • Hourly partitions for DROP PARTITION: at 1600 TPS peak, outbox gets ~1M rows/hour. Deleting individually is expensive. Instead: DROP TABLE payments_outbox_2024_01_15_14 atomically.
  • retry_count: if Kafka publish fails, increment counter. After 3 retries, alert on-call. Dead-letter queue for manually inspecting failed events.
  • Kafka retention: 30 days for payment events. Consumers replay from offset for new consumer groups. Enables: new analytics pipelines, data reprocessing, disaster recovery replay.

CQRS for reads
Write primary handles all inserts/updates. 1–5 read replicas for merchant dashboards, customer history. Elasticsearch for full-text search. Redis for real-time balances. Snowflake/BigQuery for BI.
Stateless payment service
All state in DB + Redis. Horizontal scale via Kubernetes HPA. KEDA on queue depth for outbox workers. No sticky sessions. Deploy new pods during Black Friday with zero config change.
Kafka partitioning
Partition by merchant_id to preserve per-merchant event ordering. Prevents hot partitions even if one merchant has 10x traffic. Consumers scale independently per partition group.

PgBouncer in transaction mode (not session mode). At 1600 TPS, naive connection-per-request = 1600 open Postgres connections. PgBouncer multiplexes to ~100 real connections.
09 Availability & Reliability Multi-PSP failover · Circuit breaker · Graceful degradation
PSP Health Score (recomputed every 30s)
score = 100 score -= error_rate_5min * 50 # 50 pts at 100% error rate score -= p99_latency_penalty # up to 30 pts score -= consecutive_failures * 5 # 5 pts per failure # Routing decision: if primary.score > 70: route_to(primary) elif backup.score > 70: route_to(backup) alert_oncall() else: queue_payment() # never lose the request page_oncall() # Constraint: psp_name set at AUTH, never changes # All subsequent ops must use same PSP
🟢 CLOSED — Normal Operation
All requests pass through to PSP. Error rate and latency monitored every 30s.
🔴 OPEN — Degraded
Triggered at 5% error rate over 60s. Fail-fast — no PSP calls. Route to backup PSP or queue. Opens after 30s cooldown.
🟡 HALF-OPEN — Recovering
Probe traffic only (1%). Close on success. Reopen on failure. Prevents cascading PSP re-failure on recovery.
Graceful Degradation Modes
All PSPs down → queue payments (never lose request) · Risk engine down → allow low-value transactions, block high-value · DB read replicas unavailable → read from primary (slower but available) · Analytics down → payments continue unaffected

11 Pitfall Questions Expect these at Staff level interviews
"A payment shows CAPTURED but the customer says they were charged twice. Walk me through how you'd debug this."
  • idempotency_keys table: check if two separate entries exist for different idempotency keys but same merchant+customer+amount+time. Client generated two different keys = two valid charges.
  • psp_requests table: query by customer payment method + amount + time window. Two SUCCESS rows for CAPTURE = double-capture confirmed.
  • ledger_entries: check for duplicate DEBIT/CREDIT pairs on same customer account. Double-entry violation = root cause found.
  • outbox + webhook logs: check if webhook delivered twice and merchant processed it twice on their end (merchant idempotency bug).
  • Resolution: issue refund for duplicate. Post-mortem to identify which layer lost idempotency.
"PSP returns a 200 but you never received the response. What's the payment status?"
  • UNKNOWN — set AUTHORIZATION_UNKNOWN or CAPTURE_UNKNOWN immediately. Never assume failure on timeout.
  • PSP may have successfully processed the charge. Retrying without inquiry could double-charge the customer.
  • Async reconciliation job: call PSP inquiry endpoint (stripe.retrieve(pspTransactionId)). The psp_transaction_id is in psp_requests.request_payload even if response was lost.
  • Resolve to AUTHORIZED or AUTHORIZATION_FAILED based on PSP's authoritative response. Show "Processing…" in UI throughout.
"Your reconciliation job found a $50K discrepancy. How do you investigate?"
  • Timing gap first: T+1 lag means yesterday's late captures don't appear in today's PSP file. Check if discrepancy resolves in 24h. Most "discrepancies" are timing.
  • UNKNOWN state payments: check count of payments in AUTHORIZATION_UNKNOWN or CAPTURE_UNKNOWN that haven't been reconciled. Could be sitting in limbo.
  • Currency rounding: FX conversion creates sub-cent differences. Check if $50K = many small rounding differences rather than one large mismatch.
  • Partial captures: authorized_amount vs captured_amount may differ. PSP settles captured_amount; internal records may show authorized_amount.
  • PSP file format change: check PSP changelog for CSV format updates. Column offset errors cause systematic mismatches.
  • Ownership: engineering finds root cause; finance ops decides resolution (write-off, dispute, etc.).
"How do you handle a PSP that settles in EUR but your merchants are billed in USD?"
  • Store both amounts in psp_requests and settlement_records: original_amount (USD), original_currency (USD), settlement_amount (EUR), settlement_currency (EUR).
  • FX rate locked at capture time — not settlement time. Rate stored in payment record. Merchants get predictable amounts regardless of FX movement over T+1.
  • Ledger FX entries: on settlement, create FX conversion ledger entry for the rate difference between capture-time rate and actual settlement rate. This difference is P&L for the platform.
  • Reconcile: PSP EUR settlement × locked FX rate = expected USD settlement. Difference = FX exposure. If large, consider hedging via FX derivatives.
"How do you ensure the ledger never has a non-zero balance? What if a service crashes mid-transaction?"
  • Same DB transaction: payment status update + both ledger entries + outbox event in a single BEGIN/COMMIT. If crash mid-transaction, Postgres rolls back all writes atomically.
  • No partial ledger writes possible because of single-transaction guarantee. The only risk is crash AFTER commit but BEFORE Kafka publish — that's what the outbox solves.
  • Nightly invariant check: SELECT SUM(CASE direction WHEN 'CREDIT' THEN amount ELSE -amount END) FROM ledger_entries. Must = 0. Any deviation = P0 alert immediately.
  • Compensating entries never deleted: refunds, chargebacks, FX corrections all create NEW ledger rows. Never UPDATE or DELETE existing entries. Makes auditing trivial.
"Black Friday hits and order volume spikes 50x. Your payment service starts timing out. What do you do and how do you prevent it next time?"
  • Immediate: scale out payment service pods horizontally (pre-defined HPA max). Check PgBouncer connection pool saturation — often the first bottleneck. Increase pool size if still have DB capacity.
  • Check PSP: are they also degraded? Enable backup PSP if primary health score drops below threshold. Queue payments if both PSPs unhealthy.
  • Shed non-critical load: disable real-time analytics writes, disable non-critical Kafka consumers, reduce risk engine to rules-only mode (skip ML model) to cut p99 from 150ms to 10ms.
  • Prevention: load test to 2x expected peak (not just expected). Pre-scale pods before major sales events. Cache merchant configuration in Redis to reduce DB reads. Async risk model inference with synchronous rule fallback.
