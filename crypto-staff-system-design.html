01 System Architecture Click components for detail
L1 — Client & Edge
CDN + API Gateway · scales horizontally · stateless
▶
📱Mobile App (iOS/Android)
React Native / Native · Client entry point
Biometric auth, offline balance caching, WebSocket for real-time price feed. Tokens in Keychain/Keystore only.
🚪API Gateway (Kong / Envoy)
Rate limiting · Auth · Routing
JWT validation, per-user rate limits, DDoS mitigation, TLS termination, request ID injection for tracing.
🔀BFF (Backend for Frontend)
Aggregation layer · Response shaping
Fans out to 5+ services, assembles unified portfolio response. GraphQL or REST. Reduces mobile round trips by 80%.
🔐Auth Service
OAuth 2.0 + PKCE · Step-up auth
Short-lived JWTs (15min), rotating refresh tokens, MFA enforcement, device fingerprinting, step-up for >$500 txns.
⚡WebSocket / SSE Gateway
Real-time price feed · Order status push
Persistent connections for price ticks. Server-sent events for order confirmation updates. Backed by Redis pub/sub.
L2 — Core Business Services
Microservices · Kubernetes · horizontal pod autoscaling
▶
📋Order Service
State machine · Saga orchestrator
CREATED→FRAUD_CHECK→COMPLIANCE→EXECUTING→CONFIRMING→SETTLED. Publishes events on every transition. Idempotent by order_id.
📒Ledger Service
Double-entry · Event sourcing · ACID
Append-only event log. Balance = sum of events. Per-user pessimistic locking during writes. Read replicas for queries.
💱Price / Quote Service
Aggregated pricing · Spread engine
Multi-exchange aggregation (Coinbase, Kraken, Binance). VWAP calculation. Spread varies by user tier, coin, market volatility.
🏦Fiat Payment Service
ACH · Card · Bank partner abstraction
ACH pull with 2–5 day settlement; instant buy via credit extension. Card funding via network tokens. Handles NSF, reversals.
👤User Service
Identity · KYC state · Tier limits
Source of truth for user state. Daily/monthly trading limits by tier. Feature flags by jurisdiction. Never stores raw PII.
🔔Notification Service
Push · Email · SMS · Dedup
Consumes Kafka order events. Idempotent by event_id. Retry with exponential backoff. User preference-aware fanout.
L3 — Crypto Infrastructure
HSM-backed · Air-gapped cold · Multi-sig ops
▶
🏛️Custody Service
Hot/Warm/Cold tiering · Key management
Hot (5%): online, instant. Warm (5%): HSM-locked, 30min unlock. Cold (90%): air-gapped, M-of-N quorum. Address derivation via BIP32 HD.
📡Transaction Broadcast Service
Signing · Fee estimation · Broadcast
UTXO selection (BTC), gas estimation (ETH). Signs via HSM. Multi-node broadcast. Handles stuck txns: RBF (BTC), EIP-1559 gas bump (ETH).
⛓️Blockchain Nodes
Full nodes · Mempool monitoring
Self-hosted BTC (Bitcoin Core) + ETH (Geth/Erigon). Redundant across AZs. 3rd-party (Alchemy) as fallback only. Never trust single source.
✅Confirmation Tracker
Mempool watch · Reorg handling
ZMQ subscription for BTC. ETH via block event streaming. Handles reorgs: rolls back events if block depth < finality threshold.
💧Liquidity Connector
OTC execution · Exchange routing
Smart order routing across B2C2, Galaxy, Coinbase Prime. VWAP targeting for large orders. Exposure limits per counterparty.
🧹Sweep / Consolidation
Hot→Cold · UTXO consolidation
Threshold-triggered sweep when hot wallet > $10M. UTXO fan-in to reduce future fees. Multi-sig approval workflow. Off-peak scheduling.
L4 — Compliance & Risk
Async post-processing · ML inference · Regulatory reporting
▶
🪪KYC / Identity Engine
ID verification · Sanctions screening
Jumio/Persona integration. Risk-scored tiers: Basic (photo ID), Enhanced (selfie liveness), EDD (high-risk). Re-KYC triggers on behavior changes.
🔍Transaction Monitoring (AML)
Chainalysis · Rule engine · SAR filing
On-chain graph analysis. Detects: structuring, mixer use, darknet wallet proximity. Auto-SAR generation above thresholds. 90-day lookback window.
🛡️Fraud Detection
Real-time ML · Feature store · <50ms
Gradient boosted model. Features: device, velocity, network graph, behavioral biometrics. Model served via Triton. Retrained weekly on labeled outcomes.
✈️Travel Rule Service
TRISA · VASP-to-VASP · Threshold logic
FATF Travel Rule: $3K US / €1K EU. Identifies counterparty VASP via address lookup. Falls back to manual if VASP unresolvable.
🚦Limits & Controls Engine
Daily/monthly caps · Velocity rules
Per-user daily buy/sell/withdrawal limits by KYC tier. Rolling window counters in Redis. Hard blocks, not just soft warnings.
🧑‍💼Compliance Ops Queue
Manual review · Case management
Flagged transactions routed to ops agents. SLA-driven queue (high-risk: 2hr, standard: 24hr). Full audit trail per case. Integration with JIRA/Zendesk.
L5 — Data & Platform Infrastructure
Multi-region · Active-passive replication · CDC pipelines
▶
📨Event Bus (Kafka)
Event streaming · Decoupling · Replay
Topics: order-events, ledger-events, price-feed, compliance-signals. Partitioned by user_id. 7-day retention. Schema registry (Avro).
🐘PostgreSQL (Per-Service)
ACID · PITR · Per-service isolation
Each service owns its schema. RDS Multi-AZ, automated failover <30s. Read replicas for analytics. Logical replication to Debezium for CDC.
⚡Redis Cluster
Cache · Locks · Rate limits · Pub/Sub
Price quote cache (1s TTL), idempotency keys, distributed locks (Redlock), rate limit sliding windows, WebSocket fanout via pub/sub.
🔒HSM + Vault (Key Mgmt)
Signing keys · Service secrets · MPC
AWS CloudHSM or Thales Luna. Zero plaintext key policy. HashiCorp Vault for service creds. Exploring MPC (Fireblocks) to eliminate single key risk.
📊Data Warehouse (Redshift/BQ)
Analytics · ML training · Reporting
CDC via Debezium → Kafka → S3 → DW. Powers: fraud model training, regulatory reporting, revenue analytics. dbt for transformations.
🪣Object Storage (S3)
KYC docs · Audit logs · Cold archive
Immutable buckets (Object Lock) for compliance audit trail. AES-256 encryption. 7-year retention policy. KYC documents in WORM storage.

03 Edge Cases & Failure Modes Staff-level considerations
⛓️Blockchain ReorgCRITICAL
A confirmed block gets reorganized out of the canonical chain. User's "confirmed" deposit evaporates. Our ledger credited a tx that no longer exists on chain.
Use 6-confirmation finality for large deposits. Listen for reorg events via node. Emit LEDGER_REVERSAL compensating event. Freeze user funds during reorg window. Alert on-call immediately.
💸Double Spend / Double BroadcastCRITICAL
Broadcast service crashes after signing but before recording tx_id. On restart, it could sign and broadcast the same withdrawal again — sending funds twice.
Idempotency key written to DB before broadcast. Check-then-act with distributed lock (Redlock). tx_id recorded atomically. On restart, check: "was this order already broadcast?" before signing.
🧊Stuck Transaction (Mempool)HIGH
Mempool congestion spikes (e.g. during NFT drops). Fee was too low. Transaction sits unconfirmed for hours. Users angry, funds in limbo.
BTC: Replace-by-Fee (RBF) — broadcast replacement tx with higher fee, same inputs. ETH: EIP-1559 gas bump. Monitor mempool depth and auto-bump after 30min. Alert + page on-call for large stuck withdrawals.
💰Hot Wallet DepletionHIGH
Sudden spike in withdrawals (market crash, viral influencer post) drains hot wallet faster than cold replenishment can occur. Withdrawals queue or fail.
Pre-warm: move funds from warm to hot on anomalous withdrawal velocity. Queue withdrawals vs. reject them (better UX). Automated cold-to-warm unlock pipeline with M-of-N approval via mobile push. SLA: <30min warm unlock.
📊Price Quote RaceHIGH
BTC drops 5% between when user saw the price and when their order executes. User expected $100 → 0.0025 BTC but gets less. Or price pumped and user got a better deal — arbitrage.
Quote TTL (15s). quote_id validated on execution. If expired, reject with 409 and issue fresh quote. Slippage tolerance configurable (±2% default). Large orders use TWAP execution over 5min to reduce market impact.
🔄Ledger Split-BrainCRITICAL
Network partition causes two Postgres nodes to think they're primary. Both accept writes. User balance updates are applied to different primaries. Data diverges.
RDS Multi-AZ with single-primary enforced. Patroni for HA with STONITH (Shoot The Other Node In The Head) to prevent split-brain. Read replicas are read-only. Regular reconciliation job compares ledger sum to blockchain custody balances.
🌐Exchange / OTC Counterparty FailureHIGH
Primary OTC desk (B2C2) goes offline mid-order. Fiat was debited. Crypto execution failed. Ledger now inconsistent — user has no fiat and no crypto.
Saga pattern: fiat debit is step 1, execution is step 2. If step 2 fails, compensating transaction refunds fiat. Automatic failover to secondary venue (Galaxy, Wintermute). Exposure limits per counterparty (max $10M outstanding).
🤖Fraud Ring / Sybil AttackHIGH
Coordinated fraud: 10,000 fake accounts buy BTC, cause intentional ACH failures, resulting in $1M+ in unrecoverable debt. Or: synthetic identity pass KYC using AI-generated IDs.
Device fingerprint graph — detect shared devices across "unique" accounts. Network graph: shared IPs, phone numbers, bank accounts. ML clustering to identify rings. Liveness detection (3D depth map) in KYC flow against AI-generated faces.
🔑HSM Key Compromise / LossCRITICAL
HSM device fails or is compromised. All keys inside are inaccessible or leaked. Cold wallet holding $500M in customer BTC is now at risk or unrecoverable.
Multi-party computation (MPC): key split across 3 parties, no single HSM holds full key. Geographic distribution. Key ceremony documentation. Annual recovery drills. Insurance (Lloyd's of London crypto custody policies). Proof of reserves audit.
📡Node Desync / Stale DataMEDIUM
Self-hosted Bitcoin node falls behind by 10 blocks. Confirmation tracker thinks a tx has 0 confirmations when it actually has 10. User's withdrawal appears stuck.
Monitor node block height vs. public explorers. Alert if lag >3 blocks. Fallback to 3rd-party node (Alchemy) for reads when primary lags. Never use lagged node for security-critical confirmations.
💣Flash Crash Liquidation CascadeHIGH
BTC drops 30% in 5 minutes. 100K users simultaneously hit "sell" button. Order service overloaded. Price service can't keep quotes fresh. Sells execute at wrong prices.
Circuit breaker: if order velocity exceeds 5x normal, engage queue. Price service has 500ms TTL fallback (accept slightly stale). Horizontal pod autoscaling pre-warms on volatility signal from price feed. Load shed: queue non-urgent orders.
⚖️Regulatory Change Mid-FlightMEDIUM
SEC declares a specific token a security. Overnight, trading that token is illegal in the US. Thousands of pending orders for that token in the queue.
Token-level feature flags with <1min propagation (LaunchDarkly or internal). Kill switch can halt all trading on a specific asset in seconds. Pending orders auto-cancelled with user notification. Refunds triggered via compensating transactions.
04 Scalability Design 10M DAU → 100M DAU
Order Service
Target: 50K orders/sec peak · p99 <200ms
  • Stateless pods — horizontal scale on CPU/RPS via KEDA
  • Partition Kafka topics by user_id — prevents hot shards, preserves per-user ordering
  • Async saga: only synchronous steps are fraud score + balance check. Rest is event-driven
  • Database per shard (user_id % N shards) — avoids single Postgres bottleneck
  • Write-ahead logging to Kafka before DB write — ensures event order matches intent
Ledger Service
Target: ACID at scale · No balance inconsistency
  • User-level sharding: user_id → shard. Cross-shard transfers are rare (platform wallet ↔ user)
  • Optimistic locking for read-heavy balance queries, pessimistic for writes
  • CQRS: separate write model (event log) from read model (materialized balance). Read replica serves portfolio API
  • Event sourcing enables time-travel queries — reconstruct balance at any point in history
  • Periodic snapshot (every 1000 events) to avoid replaying full history on startup
Price / Quote Service
Target: 1s freshness · millions of concurrent subscribers
  • Price aggregator is single-threaded write path; all reads from Redis cache
  • WebSocket gateway sharded by user_id prefix — 100 gateway pods, each holds 100K connections
  • Redis pub/sub broadcasts new price tick to all gateway pods simultaneously
  • Price feed degraded mode: if aggregator lags >2s, widen spread to protect against stale execution
  • Circuit breaker: if 2/3 exchanges unreachable, halt market orders, allow limit orders only
Confirmation Tracker
Target: Watch 100M addresses · <1s detection latency
  • Bloom filter for O(1) address lookup — 100M addresses in ~120MB RAM (false positive ~0.1%)
  • Verified via DB lookup on bloom filter hit — eliminates false positives before processing
  • Multiple full node connections across AZs — never single-node dependency
  • Reorg detector: compare last N block hashes every 10s against multiple nodes
  • Dedicated high-memory node for mempool tracking (mempool can be 300MB+)
Fraud Detection
Target: <50ms p99 · 50K req/sec
  • Feature computation pre-cached in Redis (velocity windows, device history) — model only does inference
  • Model served via NVIDIA Triton Inference Server — GPU-accelerated, batching, model versioning
  • Shadow mode: new model runs in parallel with production. Compare scores before promoting
  • Feedback loop: labeled outcomes (chargebacks, fraud confirmed) retrain model weekly
  • Fallback: if ML service down, rule-based fallback (velocity + device age) kicks in automatically
Multi-Region Architecture
Target: 99.99% uptime · RPO <1s · RTO <30s
  • Active-active US-East + US-West for read traffic; active-passive for writes (Ledger is single primary)
  • Ledger writes go to primary region. Replication lag <100ms to standby. Auto-failover via Route53 health checks
  • Kafka MirrorMaker2 for cross-region event replication. Consumers idempotent — safe to replay
  • Blockchain nodes in both regions — each region has independent view of chain
  • Chaos engineering (Chaos Monkey) runs weekly to validate failover paths work in practice

Sharding & Scale
  • 64 shards by user_id % 64. Each shard = independent Postgres RDS Multi-AZ instance. Cross-shard ops (platform ↔ user) use Saga, not distributed transactions.
  • Write path: always primary only. Read path: replica for balance queries (lag <100ms acceptable), primary for saga consistency checks.
  • NUMERIC(28,18) — never use FLOAT for money. Supports 10 decimal places beyond satoshi precision for future assets.
  • Idempotency UNIQUE constraint at DB level — application-level dedup is not enough. DB constraint is the final safety net against double-credit.
  • Row-level locking: SELECT FOR UPDATE on user+asset during write. Prevents concurrent balance race even across service instances.
  • Reconciliation table: nightly job computes sum of all user balances per asset vs. custody holdings. Discrepancy = PagerDuty P0.
  • CDC via Debezium: logical replication slot on primary. Events flow to Kafka → data warehouse for analytics and fraud model training.
  • Archival: partitions older than 2 years moved to Redshift. Still queryable via Athena for historical dispute resolution.

Design Decisions
  • Optimistic locking for status transitions: UPDATE orders SET status='EXECUTING', version=version+1 WHERE order_id=? AND version=? AND status='FRAUD_CHECK'. Row count=0 means concurrent update won — retry or fail.
  • Partial index on active orders is critical at scale: only ~0.1% of orders are non-terminal at any time, yet ops dashboards query them constantly. Full-table index scan would be catastrophic.
  • Redis for in-flight state: distributed lock keyed by user_id prevents concurrent orders from racing on the same user's balance. Lock TTL=30s, auto-expires on crash.
  • Separate state_transitions table — never overwrite. Append only. Lets you reconstruct what happened to any order, which is essential for P0 incidents and regulator requests.
  • Sharding by user_id — ensures all orders for a user land on the same shard. Simplifies per-user queries and saga consistency within a user scope.
  • Hot partition problem: BIGSERIAL PKs on time-ordered inserts create hot write shards. Use UUID v7 (time-ordered UUID) instead — spreads writes while preserving time-ordering for range scans.

PII separation: kyc_records in a separate DB with column-level encryption. Users table has no PII — only status/tier. GDPR erasure: encrypt PII fields with user-specific key, then destroy key. Data is "erased" without deleting financial records.
Design Decisions
  • Column-level encryption for PII via HashiCorp Vault transit engine. Encryption happens in app layer before write — DB never sees plaintext. Key rotation possible without re-writing data (re-wrap API).
  • SSN stored as bcrypt hash only — used for duplicate detection only (same SSN = existing user). Never decryptable. Salted per-user to prevent rainbow table attacks.
  • GDPR Right to Erasure: encrypt PII with user-specific DEK (Data Encryption Key). Erasure = destroy DEK. Data remains in DB (financial records can't be deleted) but is permanently unreadable.
  • S3 WORM for documents: ID scan + selfie stored with Object Lock (Compliance mode, 7-year retention). Pointer (s3_key) in DB. If DB record deleted, document still immutable on S3.
  • Audit log table: every KYC status change logged with actor_id, timestamp, reason. Append-only. Required for regulatory examination.
  • Read-heavy, write-rarely pattern: user status checked on every order (~50K/sec). Cache kyc_tier in Redis (5min TTL). Cache invalidation on status change via Kafka event.

Design Decisions
  • Bloom filter for 100M addresses: at 10 bits/element and 2 hash functions, false positive rate ~1%. On bloom hit: verify against DB. On bloom miss: skip. Prevents O(N) DB queries per block.
  • BIGINT for satoshis — 21M BTC × 10^8 satoshis = 2.1×10^15, fits in BIGINT (max 9.2×10^18). Never use DECIMAL for BTC amounts — precision errors accumulate.
  • UTXO locking: SELECT FOR UPDATE on UTXO row during fee estimation + transaction construction. Prevents two concurrent withdrawals selecting the same UTXO. Reserved status set atomically before broadcast.
  • HD path stored per address: allows key reconstruction from master seed during disaster recovery without scanning chain. Path is not secret — master seed is the secret.
  • Custody DB is air-gap adjacent: hot custody DB is online but has restricted network access. Warm/cold wallet data on separate DB with no internet connectivity — only local network from operations center.
  • Sweep audit table: every hot→cold sweep logged with: initiator, approvers (M-of-N), tx_id, amount, timestamp. Immutable. Required for SOC 2 audit.

Design Decisions
  • Bloom filter in Redis for 100M addresses: 100M addresses × 10 bits = ~120MB in Redis. False positive rate: 0.1%. On hit, verify against Postgres custody DB. On miss, discard (not our address). Eliminates nearly all DB reads for irrelevant transactions.
  • Pending transactions in Redis Sorted Set: ZADD pending:txns {seen_timestamp} {txid}. Scan with ZRANGEBYSCORE to find txns pending >30min (stuck detection). Remove on confirmation: ZREM.
  • Confirmations column vs. separate events table: UPDATE is fine here — idempotent increment. We only care about current state. But also emit Kafka event per confirmation threshold — consumers get event stream without polling DB.
  • Reorg handling query pattern: on reorg detected, query all CONFIRMED transactions at block_height >= reorg_start_height. Emit REORGED status. Ledger service has Kafka consumer that applies compensating events. Critical: must process reorg events in order — Kafka partition by chain+shard ensures ordering.
  • Sharding consideration: partition watched_transactions by chain first, then by address_hash prefix. BTC and ETH are independent workloads. Allows chain-specific scaling.

Design Decisions
  • Store token_hash not token: if refresh_tokens table is compromised, attacker gets hashes — useless without the actual token value (SHA-256 is one-way). Store SHA-256(token), validate by comparing hash of presented token.
  • JWT access tokens are stateless — not stored in DB. Validated via public key (RS256). Revocation is the challenge: maintain a Redis revocation set. SADD revoked:{jti} with TTL=token_expiry. Check on every request (fast: O(1) Redis SISMEMBER).
  • Refresh token rotation chain: replaced_by_hash creates a linked list. Detect replay attacks: if a rotated token is presented, walk the chain and revoke all tokens in the family. Signals token theft.
  • Device fingerprint_hash: composite of: OS version + screen resolution + installed app version + hardware ID. Hashed to avoid storing identifiable data. Changed fingerprint = flag as new device.
  • Redis for active session count: INCR/DECR session_count:{user_id}. Enforce max concurrent sessions (e.g. 5 devices). Oldest device session evicted on 6th login.
